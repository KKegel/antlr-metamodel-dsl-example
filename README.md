# ANTLR Metamodel DSL Template

This project is a starter template that shows how to build a textual **domain-specific language (DSL)** on the JVM using [ANTLR4](https://www.antlr.org/) and Kotlin.

The language implemented here is called **KMeta**. It is a *metamodeling language* — a language for describing the structure of data. Think of it like a schema or a class diagram written in plain text. For example, the file `example/cleaning_robot.kmeta` defines the types `Robot`, `Room`, `Obstacle`, etc. and their properties and relationships, without containing any actual robot data.

The project reads `.kmeta` files, parses them, and produces an in-memory **metamodel** object that downstream code can use to validate, generate, or interpret instance data.

> This template project is a decoupled and modified snippet of the [karpfen project language module](https://github.com/karpfenproject) 🐟

---

## Package structure

```
src/main/antlr/kmeta/     # The ANTLR grammar definition (Kmeta.g4)
src/main/java/kmeta/      # Java parser/lexer classes generated by ANTLR (checked in)
src/main/kotlin/
  meta/                   # The metamodel data classes (the "AST" of a parsed .kmeta file)
  parser/                 # Bridges ANTLR's parse tree to a simpler intermediate structure
  dsl/textual/            # Converts the intermediate structure into a Metamodel object
  dsl/functional/         # Alternative: build a Metamodel programmatically in Kotlin code
src/test/kotlin/          # Usage examples and tests
example/                  # Sample .kmeta files
```

The `bin/` and `build/` folders are generated by the build process and can be ignored.

---

## What the build.gradle file does

Gradle is the build tool. The `build.gradle` file wires up an extra step that runs **before** the normal Kotlin compilation: it invokes ANTLR to generate Java source code from the grammar file.

The key parts:

1. **`id 'antlr'`** — activates the Gradle ANTLR plugin, which knows how to call the ANTLR tool automatically.
2. **`antlr 'org.antlr:antlr4:4.13.1'`** — declares ANTLR as the code-generation tool.
3. **`antlr4-runtime`** — the small runtime library that the generated parser needs at runtime.
4. **`generateGrammarSource { arguments += ['-visitor', '-no-listener', '-package', 'kmeta'] }`** — tells ANTLR to generate *visitor* classes (the pattern used in this project to walk the parse tree) and to place all generated classes in the `kmeta` Java package.
5. **`copyAntlrSources`** — a custom task that copies the generated Java files from the build output folder into `src/main/java/kmeta/`. This makes them visible to the IDE and allows them to be committed to version control so that others do not have to re-run the generator.
6. **`sourceSets { main { java { srcDirs = ['src/main/java'] } } }`** — tells the Kotlin/Java compiler to look for Java sources in `src/main/java` (where the copies live) rather than in the build directory.

In short: `Kmeta.g4` → ANTLR generates Java → copied to `src/main/java/` → compiled together with the Kotlin sources.

## The grammar file

`src/main/antlr/kmeta/Kmeta.g4` is the single source of truth for the KMeta syntax. ANTLR reads this file and generates a complete lexer and parser from it.

The grammar describes two kinds of rules:

- **Lexer rules** (uppercase names) define the raw tokens — individual words and symbols the scanner recognises, such as the keywords `type`, `prop`, `has`, `knows`, `list`, and punctuation like `{`, `(`, `,`.
- **Parser rules** (lowercase names) define how tokens combine into meaningful structures.

A valid KMeta file is a sequence of **type definitions**:

```
type "Name" "A human-readable description" {
    prop("propertyName", "string")          // a primitive attribute
    prop("scores",  list("number"))         // a list of numbers
    has("address",  "Address")             // an embedded sub-object (composition)
    knows("friends", list("Person"))       // a reference to another object (association)
}
```

The four rule keywords map directly onto the metamodel concepts described below:
- `prop` — a simple value attribute (`string`, `number`, or `boolean`)
- `has` — owns and contains another typed object (the child cannot exist independently)
- `knows` — holds a reference to another typed object (the target exists independently)

## The metamodel AST classes (meta/)

After parsing, the DSL is represented as a tree of plain Kotlin data classes. This is the **Abstract Syntax Tree (AST)** of the metamodel — a structured, in-memory object graph that is much easier to work with than the raw parse tree.

| Class | Purpose |
|---|---|
| `Metamodel` | Top-level container. Holds all type definitions and designates one as the root type. |
| `ClassType` | One type definition. Has a name, a description, a list of `SimpleProperty`, and a list of `ClassTypeProperty`. |
| `SimpleProperty` | A primitive attribute: a name, a type (`STRING` / `NUMBER` / `BOOLEAN`), and whether it is a list. |
| `ClassTypeProperty` | A relationship to another `ClassType`: a name, whether it is a list, and the association kind (`EMBEDDED` or `LINK`). |
| `ClassTypeReference` | A pointer to a target `ClassType` by name, resolved after all types are known. |
| `SimplePropertyType` | Enum: `STRING`, `NUMBER`, `BOOLEAN`. |
| `AssociationType` | Enum: `EMBEDDED` (the child is owned here) or `LINK` (a reference to an object owned elsewhere). |

## Transformation of the ANTLR parse tree to the custom AST

Parsing a `.kmeta` file goes through several stages, each handled by a different class:

```
.kmeta text
   │
   ▼  KmetaLexer  (generated by ANTLR)
Token stream
   │
   ▼  KmetaParser  (generated by ANTLR)
Parse tree  ← low-level tree mirroring the grammar rules exactly
   │
   ▼  KmetaParseVisitor  (src/main/kotlin/parser/KmetaFileParser.kt)
KmetaFile / TypeDefinition  ← a cleaner intermediate representation
   │
   ▼  KmetaDSLConverter  (src/main/kotlin/dsl/textual/KmetaDSLConverter.kt)
Metamodel  ← the final AST ready for use
```

1. **`KmetaFileParser`** feeds the raw text into the ANTLR-generated `KmetaLexer` and `KmetaParser` to obtain a parse tree, then hands it to `KmetaParseVisitor`.
2. **`KmetaParseVisitor`** extends the generated `KmetaBaseVisitor` and walks the parse tree node by node, extracting names, descriptions, and rules into plain `TypeDefinition` data objects. This step strips away ANTLR-specific types.
3. **`KmetaDSLConverter`** iterates over the `TypeDefinition` list and maps each rule into the appropriate AST class (`prop` → `SimpleProperty`, `has` → `ClassTypeProperty(EMBEDDED)`, `knows` → `ClassTypeProperty(LINK)`). After all types are collected, a second pass resolves every `ClassTypeReference` to the actual `ClassType` object it points to. The last type in the file is treated as the root.

As an alternative to text parsing, `MetamodelBuilder` (`src/main/kotlin/dsl/functional/`) lets you construct the same `Metamodel` object directly in Kotlin code using a builder API — useful for tests or when the metamodel is generated programmatically.
